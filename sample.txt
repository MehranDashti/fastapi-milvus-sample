FastAPI is a modern, fast web framework for building APIs with Python based on standard Python type hints. It was created by Sebastián Ramírez and first released in 2018. FastAPI is built on top of Starlette for the web parts and Pydantic for the data parts. It is one of the fastest Python frameworks available, with performance comparable to NodeJS and Go. FastAPI automatically generates interactive OpenAPI documentation, making it easy for developers to test and explore APIs directly from the browser. The framework supports asynchronous programming natively using Python's async and await syntax, which allows handling thousands of concurrent requests efficiently.

FastAPI uses Python type hints extensively to validate request and response data automatically. When you define a route with type-annotated parameters, FastAPI validates incoming data, converts types, and generates documentation — all from the same type hints. This eliminates a large class of bugs that commonly appear in dynamically typed APIs. FastAPI also supports dependency injection out of the box, making it easy to share database connections, authentication logic, and other resources across route handlers. The dependency injection system is one of the most powerful features of FastAPI and allows writing highly testable and modular code.

Deploying FastAPI applications is straightforward. The recommended ASGI server is Uvicorn, which is an extremely fast server implementation. For production, it is common to run multiple Uvicorn workers behind a process manager like Gunicorn, or deploy the application as a Docker container behind an Nginx reverse proxy. FastAPI applications can also be deployed on serverless platforms like AWS Lambda or Google Cloud Run using adapters like Mangum.

Milvus is an open-source vector database built specifically for AI applications and similarity search at scale. It was created by Zilliz and first released in 2019. Milvus is designed to store, index, and manage massive embedding vectors generated by deep neural networks and other machine learning models. It can handle billions of vectors with millisecond-level query latency, making it suitable for production AI systems. Milvus is written in Go and C++ for high performance and supports deployment on a single machine or as a distributed cluster across many nodes.

Milvus supports multiple index types optimized for different use cases. The HNSW (Hierarchical Navigable Small World) index provides the best recall and speed tradeoff for most production use cases. It builds a multi-layered graph structure where each node connects to its nearest neighbors, allowing search to navigate quickly from coarse to fine granularity. The IVF_FLAT index divides the vector space into clusters using k-means and searches only the nearest clusters during query time, reducing computation at the cost of some recall. The SCANN index is optimized for billion-scale datasets and uses product quantization to compress vectors and speed up distance calculations significantly.

Milvus organizes data into collections, which are equivalent to tables in a relational database. Each collection has a schema defining the fields and their data types. Every collection must have exactly one vector field, which stores the embedding vectors. Other fields are scalar fields used to store metadata like document titles, timestamps, categories, or source URLs. Milvus supports filtering search results using scalar fields, allowing queries like "find the 10 most similar vectors where category equals technology and created_at is greater than last week."

Milvus uses a segmented storage architecture. When data is inserted, it first goes into a growing segment in memory, which is immediately searchable. When a growing segment reaches a configured size threshold, it is sealed and flushed to persistent storage as a sealed segment. This design makes writes fast and reads efficient. Milvus also supports data partitioning, which allows splitting a collection into logical partitions based on a partition key. Searching within a specific partition is faster than searching the entire collection because Milvus skips irrelevant partitions entirely.

OpenAI provides powerful APIs for both embeddings and language model completions. The Embeddings API converts text into dense numerical vectors that capture the semantic meaning of the input. The text-embedding-3-small model produces 1536-dimensional vectors and is optimized for efficiency and cost. The text-embedding-3-large model produces 3072-dimensional vectors with higher accuracy for demanding applications. Embeddings are widely used in semantic search, clustering, classification, anomaly detection, and recommendation systems.

The OpenAI Chat Completions API provides access to powerful language models like GPT-4o and GPT-4o-mini. These models can understand and generate natural language, answer questions, summarize documents, write code, translate languages, and perform many other tasks. The API accepts a list of messages with roles (system, user, assistant) and returns a generated response. The system message is used to set the behavior and persona of the assistant. GPT-4o-mini is a cost-efficient model suitable for most everyday tasks, while GPT-4o provides higher accuracy for complex reasoning tasks.

Retrieval Augmented Generation, commonly known as RAG, is a technique that combines the power of vector search with large language models. The core idea is to ground LLM responses in external knowledge by retrieving relevant documents at query time and providing them as context. This approach solves a fundamental limitation of language models: their knowledge is frozen at training time and they cannot access private or up-to-date information. With RAG, you can build question-answering systems over your own documents without fine-tuning a model.

The RAG pipeline has two main phases. The ingestion phase processes documents offline: documents are loaded from files or databases, split into smaller chunks, converted to embedding vectors using an embedding model, and stored in a vector database like Milvus. The retrieval phase happens at query time: the user's question is embedded using the same model, the vector database finds the most semantically similar chunks, and those chunks are passed to the language model as context along with the original question. The language model then generates an answer grounded in the retrieved context.

Chunking strategy is one of the most important design decisions in a RAG system. Chunks that are too small may not contain enough context for the language model to generate a good answer. Chunks that are too large may dilute relevance — the vector represents an average of all content in the chunk, making it harder to find precise matches. A common approach is to use chunks of 256 to 512 tokens with an overlap of 20 to 50 tokens between consecutive chunks. The overlap ensures that information at chunk boundaries is not lost and appears fully in at least one chunk.

Redis is an in-memory data structure store that supports strings, hashes, lists, sets, sorted sets, bitmaps, and more. It is commonly used as a cache, message broker, and session store. Redis persistence is configurable: RDB snapshots save the dataset to disk at intervals, while AOF logs every write operation for durability. Redis Cluster allows horizontal scaling by sharding data across multiple nodes automatically. Redis Sentinel provides high availability by monitoring master and replica nodes and performing automatic failover when the master becomes unavailable.

MongoDB is a document-oriented NoSQL database that stores data in flexible JSON-like documents called BSON. Unlike relational databases, MongoDB does not require a fixed schema, allowing documents in the same collection to have different fields. MongoDB supports rich query operations including filtering, projection, aggregation pipelines, and geospatial queries. It scales horizontally through sharding, distributing data across multiple nodes based on a shard key. MongoDB Atlas is the managed cloud offering that handles infrastructure, backups, and scaling automatically.

Docker is a platform for building, shipping, and running applications in containers. A container is a lightweight, isolated environment that packages an application with all its dependencies, ensuring consistent behavior across development, staging, and production environments. Docker images are built from Dockerfiles, which contain instructions for assembling the image layer by layer. Docker Compose is a tool for defining and running multi-container applications using a YAML configuration file. Kubernetes is the industry-standard container orchestration platform for deploying, scaling, and managing containerized applications in production clusters.

Python virtual environments allow isolating project dependencies from the system Python installation. The venv module creates a self-contained directory with its own Python interpreter and site-packages. This prevents version conflicts between projects and makes dependency management reproducible. Tools like pip, pip-tools, and Poetry manage package installation and lock files. A requirements.txt file lists all direct and transitive dependencies with pinned versions, ensuring that the exact same environment can be recreated on any machine.

RabbitMQ is an open-source message broker that implements the Advanced Message Queuing Protocol (AMQP). It decouples producers and consumers by routing messages through exchanges and queues. RabbitMQ supports multiple exchange types: direct exchanges route messages to queues with a matching routing key, topic exchanges use wildcard patterns, fanout exchanges broadcast to all bound queues, and headers exchanges route based on message headers. Dead letter exchanges capture messages that cannot be delivered or are rejected by consumers, allowing error handling and retry logic without losing messages.

Microservices architecture decomposes a large application into small, independently deployable services that communicate over well-defined APIs. Each service owns its own data store and can be developed, deployed, and scaled independently. This approach improves fault isolation — a failure in one service does not bring down the entire application. However, microservices introduce distributed systems challenges such as network latency, partial failures, data consistency across services, and the need for service discovery and load balancing. Common patterns for handling these challenges include the circuit breaker pattern, the saga pattern for distributed transactions, and event sourcing with CQRS.

Domain Driven Design (DDD) is a software development approach that centers the design around the core domain and domain logic. It introduces a rich vocabulary including entities, value objects, aggregates, repositories, domain services, domain events, bounded contexts, and ubiquitous language. An entity is an object with a unique identity that persists over time. A value object has no identity and is defined entirely by its attributes. An aggregate is a cluster of entities and value objects with a single root entity that enforces consistency boundaries. Bounded contexts define explicit boundaries within which a particular domain model applies, preventing ambiguity when the same term means different things in different parts of a large system.
